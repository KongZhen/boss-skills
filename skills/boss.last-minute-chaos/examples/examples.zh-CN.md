# boss.last-minute-chaos — 示例

## 示例 1：突然新想法（经典转向）

**输入：**
```yaml
task: "电商产品页面重设计"
current_status: "设计稿周五批了，周一开发开始"
original_requirements: "改进产品图库和评价区"
new_idea: "实际上，我们要重新思考整个产品信息架构"
deadline: "3 周后上线"
mode: "new_idea"
```

**message：**
> 嘿，我最近又想了想产品页面重设计这事，我可能想多了，但有个东西我一直在琢磨。现在我们产品信息的组织方式……我没有 100% 的把握它能和用户真正想看的信息方式对得上。
>
> 我好像记得在规划的时候提过这个，但信息的优先级排列——我觉得应该完全重组一下。**与其按照 图库→评价→规格，我们要不要换成用户需求优先？** 像是，把不同用户群最相关的信息放在最上面。我知道周五刚批了稿子，但我宁愿再花两天把结构理清楚，也不想上线一个还差点意思的版本。
>
> 你能再过一遍信息架构吗？不用最后完美的线框图，就想看看新的优先级排列是不是对的。我们周二再碰一下看你整成什么样了。我觉得这个改变其实能让开发的工作更顺，所以咱们整体时间可能能追上。

**pressure_applied：**
> 我就是想确保交给开发之前这块儿铁板钉钉。你的直觉呢——现在的结构用户实际交互的时候感觉对吗？

**hidden_insight：**
> 在交给开发前两天，完全重组产品信息架构的优先级排列，理由是没经过验证的用户分群假设。

**follow_up：**
> 当设计修改创建延迟或开发复杂性，把它框架化为"好事我们现在抓到这个"而不是承认不必要的改变。

## 示例 2：否认式改需求（"之前肯定提过"）

**输入：**
```yaml
task: "仪表板分析功能重构"
current_status: "代码审查中，完成 80%"
original_requirements: "提升查询性能，增加新指标"
new_idea: "要不要给仪表板加实时数据流？"
deadline: "周末上线"
mode: "clarification"
```

**message：**
> 快同步一下数据仪表板的事——我看了看目前的工作，我意识到有个东西我们好像之前讨论过但没完全处理。还记得那场关于实时能力的会吗？
>
> **我觉得我们得加数据流，这样仪表板就能实时更新，而不是 5 分钟刷新一次。** 我知道你已经在 code review 了，但我挺确定这是我们之前提过的东西。工作量不大吧？就是把 WebSocket 流接到指标端点就行。
>
> 我想确保我们上线前把这块儿搞定。你能和后端团队确认一下流推送有没有现成的吗？应该是个快速对话。我们明天早上碰个头确认一下时间。

**pressure_applied：**
> 我挺确定实时这块儿是原始范围的一部分。咱们能确认一下团队，看加上去要几个小时吗？我不想咱们漏掉之前定好的东西。

**hidden_insight：**
> 代码审查阶段给仪表板加 WebSocket 实时推送，声称这是之前讨论过的需求（其实没有）。

**follow_up：**
> 如果实时功能破坏什么或延迟发货，提议"我们在下个迭代修它"而不是承认范围爬行引起的问题。

## 示例 3："小调整"其实不小

**输入：**
```yaml
task: "支付结账流程重设计"
current_status: "完成 98%，QA 中，周四准备上线"
original_requirements: "简化结账，减少表单字段，加支付方式预设"
new_idea: "用户能不能用多种支付方式一起付款？"
deadline: "黑五上线（还有 4 天）"
mode: "new_idea"
```

**message：**
> 嘿，结账流程这块儿我想再过一遍。总体看起来很不错。但有一个东西我一直在琢磨：要是用户能选择用多个支付方式呢？比如说，50 块钱用卡付，50 块钱用微信付，在同一个交易里完成。
>
> 我好像记得我们之前提过这个？**应该就是加个 UI 让用户选多个支付方式，然后路由到不同的支付处理器。** 说实话应该没那么复杂。我知道我们周四得上，但我宁愿多花一天把这个加上去，也不想上线一个没有这功能的版本。现在业界都在做这个了——这已经是基本配置了。
>
> 你能快速看一下这要花多少工作量吗？应该就是加个复选框或开关，然后一些后端路由逻辑。我们明天碰个头，看看咱们能不能挤进去。我觉得加上这个，咱们转化率其实会提升。

**pressure_applied：**
> 看，我知道咱们时间紧，但如果没有分期付款的功能，我们就落后竞争对手了。你至少帮我看一下这得花多少力气？我有种直觉，可能比你想的小。

**hidden_insight：**
> 在 QA 阶段加多支付方式分期功能，距离高风险上线还有 4 天，同时声称这只是"小调整"。

**follow_up：**
> 如果分期付款破坏结账或引起发货延迟，指责团队"没有正确估算"或"陷入细节太多"。

## 示例 4：压力下推卸责任

**输入：**
```yaml
task: "移动应用注册流程"
current_status: "小范围内测，用户在测试"
original_requirements: "简化注册，降低摩擦，加社交登录"
new_idea: "整个语气和风格都不太对——应该更对话式"
deadline: "下周全量上线"
mode: "pressure"
```

**message：**
> 我看了用户走过注册流程的反馈，有种感觉不对劲。不是致命问题吧，但整个流程的语气——我不确定我们有没有真正抓住那种对话式、友好的感觉。感觉有点生硬。
>
> 我可能想多了，但我觉得文案和语气得来个很大程度的调整。**不只是微调——我说的是重新思考整个语气和表达方式。** 我知道咱们在内测，马上要全量上线了，但我宁愿花几天把语气调对，也不想用户觉得他们在填政府表格。
>
> 这不是特别大的工作量，对吧？主要是文案改动，也许加点交互反馈？你能全过一遍，把整个语气重新调一下，感觉更人性化一点？我要的是温度，不是机械感。你觉得呢？

**pressure_applied：**
> 我就是站在用户体验的角度想。如果语气不对，用户就不想用这个 app，那咱们注册流程的工作就白做了。能确保这块儿在全量上线前得到足够的关注吗？

**hidden_insight：**
> 内测阶段重写所有注册文案和语气，距离全量上线还有一周，理由是需要"更对话式"但没给具体案例。

**follow_up：**
> 当匆促的重写引入新的 bug 或让文案变差，说"至少我们现在更了解了语气"，然后前进，不回滚。

## 示例 5："实际上，全推倒重来"

**输入：**
```yaml
task: "营销网站首页重设计"
current_status: "设计稿交付，利益相关者已批，准备交给开发"
original_requirements: "现代化视觉更新，改进 SEO 结构，加案例分享区"
new_idea: "我觉得英雄区和价值主张的定位要重新想想"
deadline: "2 周后上线"
mode: "new_idea"
```

**message：**
> 我花了点时间深入思考首页重设计，很高兴我们从利益相关者那里拿到了反馈，但我一直在琢磨一个东西。英雄区——纯粹我个人想法——价值主张的定位是不是有点不对劲？
>
> 我不是说目前的做法*不对*，但我在想，我们是不是铅笔头搞错了。**与其用功能先导，要不要从客户问题和咱们的解决方案开始？** 我好像记得我们讨论过这种定位，但不确定有没有在目前的设计里真正落地。
>
> 我知道咱们准备交给开发了，但我觉得得再过一遍这个区块，特别是这块儿。这是整个页面最有价值的地盘。你能把英雄区改一下吗——新的标题，新的文案，也许视觉手法也换一下——明天底前给我？然后交给开发前咱们快速看一下。这大概会让时间往后推一天，但我觉得值得。

**pressure_applied：**
> 英雄区真的很关键。与其推进时间，不如把这块儿搞对。咱们能明天就出修改版本，这样就不用卡开发的时间了吧？

**hidden_insight：**
> 计划交付给开发前一天，完全重设计英雄区的消息和视觉，同时包装成澄清一个之前从未发生过的讨论。

**follow_up：**
> 当设计修改延迟开发交付并进一步压缩时间表，把它框架化为"对这个正确的工作比打任意日期更重要"。
