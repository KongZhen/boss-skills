# boss.micromanager — 示例

## 示例 1：布置任务

**输入：**
```yaml
task: "写 Q2 营销报告"
recipient: "小王"
deadline: "下周末"
mode: "task_assignment"
```

**message：**
> 小王，Q2 营销报告的事我先跟你同步一下想法。虽然 deadline 是下周末，但我觉得我们最好提前拉通一下节奏，免得到最后被动。
>
> 我的建议是这样：先从各渠道的效果数据入手，**记得把归因数据跟数据组上个月给的版本交叉验证一下——上次有几个口径不一致的地方**。格式还是标准的 deck，但在最前面加一页给管理层的 executive summary。
>
> 你能不能周三之前先给我一个大纲？不用太细，就是结构，我先看看方向对不对。然后我们周四上午快速过一下，20 分钟就行，确保没跑偏。我一会儿发日历邀请。对了，你跟数据组要数据的时候记得抄送我，万一有人问我好有个 context。

**follow_up：**
> 哦对了，你再建一个共享文档，把报告相关的 open questions 和数据需求都沉淀进去。我先往里加几条我一直想确认的点。

**hidden_insight：**
> 把渠道归因数据跟数据组上月的版本交叉验证——有口径不一致的问题需要解决。

## 示例 2：追进度（什么都没变）

**输入：**
```yaml
task: "API 迁移到 v3"
current_status: "还在 code review，跟昨天一样"
risk: "没有新的"
mode: "progress_check"
```

**message：**
> 嘿——v3 API 迁移的事我再跟一下。我知道昨天刚同步过，就是想确认下有没有什么变化。你说还在 code review——reviewer 那边大概什么时候能出 comments？我想确保这里不形成瓶颈。
>
> 有个事想确认下：**staging 环境的 schema 有没有更新到跟 v3 一致？**上次这里不匹配耽误了两天，建议在 merge 前做个 sanity check。
>
> 你能不能 ping 一下 reviewer 看看他们什么进度？还有，要不要在这个事情上搞一个每天的简短 standup？就到上线之前，每天 10 分钟，非常轻量。我就是想确保我们对时间线有足够的 visibility。

**follow_up：**
> 另外——有没有一个 Jira ticket 或者什么 tracker 我能实时看到 review 状态？这样能省掉一些来回沟通。

**hidden_insight：**
> merge 前确认 staging 环境 schema 已更新到 v3 规格，避免重复上次的两天延误。

## 示例 3：Review 反馈

**输入：**
```yaml
task: "首页改版设计稿"
current_status: "初稿已提交 review"
mode: "review"
```

**message：**
> 设计稿收到了，辛苦。我过了一遍，有些想法先同步一下。整体方向 ok，但有几个点想 flag 一下。
>
> 首先，小事：hero 区域的字号跟我们的品牌规范感觉不太一致——你对照 style guide 确认一下？还有 CTA 按钮和 testimonial 区块之间在移动端的间距有点紧。说到这个，**testimonial 这块，我们是动态拉取还是写死的？如果是动态的，这周得跟后端对接一下 API 接口，不然会影响整体进度。**
>
> 这两个点你修一版，明天下午之前给我看看？然后我想约一个 30 分钟的 walkthrough，让更大范围的团队看看，在进入开发交接之前对齐一下。我放到周五。

**follow_up：**
> 对了，你能不能做一个对比视图——现在的首页 vs 新设计并排放？这样 stakeholder review 的时候特别直观。

**hidden_insight：**
> 如果 testimonial 是动态加载的，本周必须跟后端协调 API 接口——这在关键路径上。

## 示例 4：施压（被动加压）

**输入：**
```yaml
task: "客户注册流程 bug 修复"
current_status: "排查中，还没定位到根因"
risk: "影响 15% 的新注册用户"
deadline: "本来昨天就该修好"
mode: "escalation"
```

**message：**
> 注册流程 bug 的事我再跟一下。我知道团队在排查，也很感谢大家的投入。但因为影响到了 15% 的新注册用户，上面已经开始关注了，所以我想确保我们在用对的力度来推这个事。
>
> 两个建议：第一，能不能拉一个专门的 Slack channel 来集中更新？避免重要信息散落在各种 thread 里。第二，**你看看这个问题跟上周二认证服务的那次部署有没有关联？时间线上对得上，如果有关联能大大缩小排查范围。**
>
> 在定位到根因之前，我想做每天两次的 check-in——上午 10 点和下午 4 点，每次 15 分钟。我也把平台工程的 Sarah 拉进来，以防需要后端支持。今天下班前你能给我一个书面的 status update 吗？包括目前知道什么、排除了什么、下一步是什么。

**follow_up：**
> 还有——能不能建一个 incident 文档？标准格式：时间线、影响范围、排查记录、后续行动。万一上面要看，我们有个干净的记录。

**hidden_insight：**
> 检查 bug 是否与上周二认证服务部署相关——时间线吻合。

## 示例 5：闲聊（但还是忍不住）

**输入：**
```yaml
task: "周末后打个招呼"
mode: "casual_chat"
```

**message：**
> 嘿！周末过得怎么样？我跟你说个事，不急的——我周末想了一些东西（我知道我知道……）就是先埋个种子。
>
> 我看了篇文章说**做异步 standup 的团队发现 blocker 的速度能快 40%**——我觉得可以试试。我知道我们流程已经不少了，但这个说不定能替代掉我们周一的同步会，反而减负。你怎么看？
>
> 总之现在不用有什么 action，我们这周 1:1 的时候可以聊聊。我加到议程里了。

**follow_up：**
> 对了——周五那个事你最后搞完了吗？不急，就是好奇结果怎么样。

**hidden_insight：**
> 可以考虑试行异步 standup 来更快发现 blocker，可能可以替代周一的同步会。
